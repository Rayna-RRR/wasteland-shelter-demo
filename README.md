# 《废土避难所值班长》原型

## 项目简介

《废土避难所值班长》是一个基于原生微信小程序、Flask 和 SQLite 搭建的轻量废土避难所管理原型，英文项目名为 `wasteland-shelter-demo`。

这个项目的目标不是做一个完整商业游戏，而是用较小的代码规模，把一条可以运行、可以复盘、也更接近玩家视角的系统循环表达清楚。当前原型围绕避难所资源运转、幸存者管理、值勤派遣、条件式补给触发和日志记录展开，并已经补上首次进入流程，让项目从单纯的系统展示更进一步变成了一个轻量可玩原型。

## 核心玩法原型

当前核心循环为：

```text
招募 -> 派遣 -> 资源变化 -> 状态反馈 -> 商业化触发 -> 日志记录
```

玩家通过招募获得幸存者，再把幸存者派去执行外出搜集、发电维护、库存炊事或夜间守卫等任务。后端根据任务结算资源变化，同时更新幸存者疲劳和健康。资源或队伍压力达到条件时，会触发救急型的应急补给协议，相关行为也会写入日志，便于后续复盘。

## 玩家侧视角 / onboarding + 游戏初始化

这一阶段新增了首次进入流程，用来补足玩家视角，让原型不再只是“打开就能操作的系统页面”，而是更像一局真正开始的避难所管理体验。

首次进入时，玩家需要先完成避难所登记，包括：

- 输入避难所代号
- 输入指挥官名字
- 选择开局难度：稳健 / 标准 / 极端
- 阅读一段简短的废土背景文案

确认后，前端会调用后端初始化接口，按难度写入开局资源，并保存当前玩家身份信息。完成初始化后，首页会展示：

- 避难所代号
- 指挥官
- 当前难度

未初始化时，主玩法页面和后端主流程接口都会被拦截，避免绕过 onboarding 直接进入游戏流程。已完成初始化后，再次进入不会重复展示 onboarding。

### 难度对应开局资源

当前代码中三种难度会覆盖开局资源。这里建议以仓库内实际实现为准；如果你准备继续下调招募券，可以同步修改这一节。

- 稳健：food 120 / power 110 / materials 90 / premium_currency 60
- 标准：food 100 / power 100 / materials 70 / premium_currency 40
- 极端：food 70 / power 75 / materials 45 / premium_currency 20

这一步的意义不只是多了一个表单，而是让项目从“系统原型”变成了“有开局、有身份、有难度差异的轻量可玩原型”。

## 当前已实现

### 幸存者招募

- 支持单次招募幸存者
- 幸存者具有姓名、稀有度、职业和内部性格来源字段
- 已拥有的同名幸存者再次出现时，会转化为材料补偿
- 招募结果会生成入队记录，并写入招募日志

### 值勤派遣

- 支持选择幸存者并派遣到不同值勤任务
- 当前值勤类型包括外出搜集、发电维护、库存炊事和夜间守卫
- 每次值勤会结算食物、电力、材料变化
- 值勤任务有基础运转成本，部分任务会消耗资源
- 每隔一定次数会触发避难所例行维护成本

### 幸存者状态管理

- 幸存者动态状态来自 `fatigue` 和 `health`
- 当前状态会派生为“状态良好”“疲惫”“状态不稳”“重伤”等动态标签
- 疲劳过高或健康过低会影响值勤收益
- 高风险任务可能造成健康下降
- 静态展示已拆成三层：
  - 特质：简短的静态性格词
  - 作风：描述幸存者平时如何处理工作
  - 档案：一句废土氛围的人物记录

### 应急补给协议

应急补给协议是条件触发的救急型设计，不是常驻商城。

当前会根据避难所压力判断是否触发补给协议。压力来源包括食物压力、电力压力、材料压力、多项资源压力、队伍状态压力和近期行动效率下滑。

补给协议出现后，会记录为“已触发”；玩家暂时不处理时，会记录为“已暂缓”；玩家购入后，会记录为“已购入”。

购入应急补给协议后，会补充招募券、食物、电力和材料，并恢复部分幸存者状态。这个设计主要用于表达“压力出现时的救急补给”，而不是完整商店、礼包系统或支付闭环。

本地开发环境下已支持“补给测试状态重置”，方便反复验证补给触发、暂缓和购入流程。

### 日志系统

- 招募日志：记录幸存者姓名、稀有度、职业和时间
- 值勤日志：记录幸存者、值勤类型、结果文本、资源变化和时间
- 应急补给日志：记录补给协议已触发、已暂缓、已购入、触发原因、资源快照、幸存者数量和时间

这些日志用于复盘系统循环，让资源压力、玩家操作、状态变化和补给触发之间的关系更清楚。

### 前端 UI 收口

- 已实现微信小程序首页、招募页、值勤页和日志页
- 幸存者卡片保留动态状态徽章，并将静态信息拆成“特质 / 作风 / 档案”
- 招募和值勤页面已经围绕当前玩法循环做了基础视觉收口
- 当前 UI 以可运行、可展示、可调试为目标，没有接入大型美术资源管线

## 技术结构

- 前端：原生微信小程序
- 后端：Python 3.9、Flask、flask-cors
- 数据库：SQLite，使用 Python 标准库 `sqlite3`
- 本地开发：macOS、微信开发者工具、VS Code

## 接口概览

常用接口包括：

- `GET /api/status`：检查后端是否运行
- `GET /api/init/status`：读取当前是否已完成游戏初始化
- `POST /api/init`：登记避难所代号、指挥官和难度，并写入开局资源
- `GET /api/resources`：查看当前资源
- `POST /api/gacha`：执行一次幸存者招募
- `GET /api/survivors`：查看幸存者列表
- `POST /api/duty`：派遣幸存者执行值勤
- `POST /api/rest`：让幸存者休整
- `GET /api/gacha-logs`：查看招募日志
- `GET /api/duty-logs`：查看值勤日志
- 应急补给协议状态接口：查看当前是否已触发补给协议
- 应急补给协议日志接口：查看补给协议已触发、已暂缓和已购入记录

说明：

- 未初始化时，除初始化相关接口外，主玩法接口会返回拦截结果
- 已完成初始化后，招募、值勤、日志和补给相关流程按正常逻辑运行
- 本地调试应急补给协议时，可以使用“补给测试状态重置”接口清理补给记录，便于重新验证流程

## 本地运行方式

初始化或修复本地 SQLite 数据库：

```bash
python backend/init_db.py
```

启动 Flask 后端：

```bash
python backend/app.py
```

后端默认运行在：

```text
http://localhost:5001
```

然后在微信开发者工具中打开 `miniprogram/` 目录，并确认 `miniprogram/config.js` 中的接口地址指向本地 Flask 后端。

首次启动时，首页会先进入 onboarding。填写避难所代号、指挥官名字并选择难度后，才会正式进入原有首页资源、招募、值勤和日志流程。完成初始化后，再次进入不会重复展示 onboarding。

## 当前完成度

当前版本已经完成一条可运行的轻量系统循环，并补上了玩家侧首次进入流程，适合作为“轻量游戏原型 + 系统设计表达 + 作品集项目”展示。

当前版本适合展示的点包括：

- 用轻量代码完成了一个完整的管理循环
- 后端结算逻辑清楚，资源、状态和日志能互相解释
- 幸存者有动态状态和静态人物展示层，便于表达角色反馈
- 应急补给协议体现了条件式商业化触发，而不是简单常驻入口
- 微信小程序前端已经能串起 onboarding、招募、派遣、日志查看等核心页面
- 项目已经从单纯的系统原型推进到更接近玩家视角的可玩原型

## 后续优化方向

- 补充更多幸存者和职业差异
- 扩展更多值勤类型和风险事件
- 增加更清晰的数值调试面板
- 优化日志筛选和复盘体验
