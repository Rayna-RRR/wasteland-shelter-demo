# 废土避难所值勤管理器

这是一个面向作品集展示的系统驱动型避难所管理原型。前端使用原生微信小程序，后端使用 Flask + SQLite。

项目定位偏向游戏系统设计、运营玩法验证和轻量 R&D 支持。它一个有系统骨架的游戏原型：确定性的资源循环、幸存者状态管理、基于玩家压力感知的商业化触发，以及可供复盘的全链路日志。

## 当前原型范围

目前已经实现的内容包括：

- 通过类 gacha 流程招募幸存者。
- 具名幸存者唯一化处理，重复招募会转为材料补偿。
- 使用可点击幸存者卡片进行值勤分配。
- 避难所资源：食物、电力、材料、招募券。
- 幸存者状态管理：疲劳和健康。
- 基于幸存者 `mood + role` 的性格标签、档案短句和当前状态标签。
- 基于本地规则模板生成的值勤结果文本。
- 来自资源短缺、避难所维护、电力短缺、队伍状态恶化的压力系统。
- 由真实避难所压力触发的应急救援补给逻辑。
- 关闭补给卡后，对当前短压力周期进行轻量 suppress，避免立即反复弹出。
- 根据压力来源变化的应急补给标题、说明和紧急标签。
- 招募、值勤、应急补给行为的日志记录。

## 核心循环

1. 在招募页抽取幸存者。
2. 查看幸存者的性格、职业和当前状态。
3. 把幸存者分配到外出搜集、发电维护、库存炊事或夜间守卫等值勤任务。
4. 后端确定性结算资源变化，以及幸存者疲劳 / 健康变化。
5. 应用值勤运转成本和周期性避难所维护成本。
6. 检测食物短缺、电力短缺、材料压力、行动低效或队伍状态压力。
7. 在压力成立时，展示一个应急救援式补给 offer。
8. 在日志页记录并查看招募、值勤和 offer 行为。

## 幸存者状态层

幸存者目前使用一个轻量但可见的状态模型：

- `name`、`rarity`、`role`、`mood` 构成角色身份。
- `mood + role` 派生出展示用的性格标签和避难所主题档案短句。
- `fatigue` 和 `health` 会影响值勤产出和行动风险。
- `状态良好`、`疲惫`、`状态不稳`、`重伤` 等派生状态标签让队伍情况更容易阅读。

这个状态层有意保持简单，用来支撑角色感和系统反馈，而不是把原型扩展成复杂 RPG 模拟器。

## 避难所压力与应急补给逻辑

当前的应急补给是一个单一救援式 offer，不是完整商店系统。

目前支持的压力信号包括：

- 食物压力。
- 电力压力和电力短缺。
- 材料压力。
- 多项资源同时接近警戒线。
- 队伍疲劳 / 健康压力。
- 连续低效率值勤结果。

当 offer 出现时，它的标题、副标题和紧急标签会根据当前 `trigger_reason` 选择对应文案。玩家关闭 offer 后，同一压力原因和同一资源快照下不会立刻再次出现；如果压力升级或资源状态发生变化，offer 可以重新出现。

offer 的奖励结构保持稳定且简单：

- 招募券。
- 食物。
- 电力。
- 材料。
- 小幅幸存者状态恢复。

这样可以把商业化层表达为“运营侧救援机制”，而不是完整现金商店。

## 日志与分析价值

原型记录三类日志：

- 招募日志：幸存者姓名、稀有度、职业和时间。
- 值勤日志：幸存者、值勤类型、结果文本、资源变化和时间。
- 应急补给日志：exposed / closed / purchased 行为、触发原因、资源快照、幸存者数量和时间。

这些日志让系统循环可审计。查看者可以追踪资源压力、玩家操作、offer 曝光、关闭或购买行为之间的关系。

## 可选 Agent / LLM 方向

当前核心模拟不依赖 LLM。

后续可以扩展的方向：

- 接入轻量中文 LLM 或 agent 层，只用于可选特殊事件文案。
- 保持资源结算、值勤结果、幸存者状态变化和 offer 资格判断为确定性逻辑。
- 把生成文本当作表现层增强，而不是经济系统或玩法结算的事实来源。

## 技术栈

- 后端：Python 3.9、Flask、flask-cors、SQLite。
- 前端：原生微信小程序。
- 数据存储：`backend/data/` 下的本地 SQLite 数据库。

## 后端接口

当前主要接口：

- `GET /api/status`
- `GET /api/resources`
- `POST /api/gacha`
- `GET /api/survivors`
- `POST /api/duty`
- `POST /api/rest`
- `GET /api/gacha-logs`
- `GET /api/duty-logs`
- `GET /api/emergency-offer/state`
- `POST /api/emergency-offer/expose`
- `POST /api/emergency-offer/close`
- `POST /api/emergency-offer/purchase`
- `GET /api/emergency-offer/logs`

## 本地运行说明

初始化或修复本地 SQLite 表结构：

```bash
python backend/init_db.py
```

启动后端：

```bash
python backend/app.py
```

然后在微信开发者工具中打开 `miniprogram/` 目录，并确认前端 API 地址指向本地 Flask 后端。

## 一句话总结

这个原型的价值在于展示一条可信的系统链路：角色招募、值勤分配、资源压力感知、补给触发，以及能解释系统行为的全链路日志。
它不覆盖完整的战斗、建筑或合成系统，支付也是模拟实现——但经济循环和触发逻辑是认真设计过的，可以被当作系统设计意图来读。
