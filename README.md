# 《废土避难所值班长》原型

## 项目简介

《废土避难所值班长》是一个基于原生微信小程序、Flask 和 SQLite 搭建的轻量废土避难所管理原型，仓库名为 `wasteland-shelter-demo`。

这个项目当前的目标不是做一个完整商业游戏，而是先用较小的代码规模，把一条**可运行、可演示、可复盘**的系统循环做出来，并让它逐步从“系统原型”推进到“有明确威胁弧线的单局制小游戏 MVP”。

目前已经完成的部分包括：玩家侧 onboarding、幸存者招募、值勤派遣、状态消耗、条件式应急补给和日志复盘，以及围绕演示路径做过的一轮前端收口。

## 当前版本的核心循环

当前原型围绕这条循环展开：

```text
onboarding / 初始化
-> 招募幸存者
-> 派遣值勤
-> 资源变化与状态消耗
-> 压力触发应急补给协议
-> 日志复盘
```

玩家进入避难所后，需要先完成初始化登记，再通过招募获得幸存者，把角色投入不同值勤任务，承受资源变化和状态代价。当避难所资源与队伍状态进入压力区间时，系统会触发一次救急型应急补给协议；所有关键行为都会进入日志，方便回看整条链路。

## 已实现内容

### 1. 玩家侧 onboarding / 游戏初始化

首次进入时，玩家需要先完成避难所登记，包括：

- 输入避难所代号
- 输入指挥官名字
- 选择开局难度：稳健 / 标准 / 极端
- 阅读一段简短的废土背景文案

确认后，前端会调用后端初始化接口，按难度写入开局资源，并保存当前身份信息。完成初始化后，首页会展示：

- 避难所代号
- 指挥官
- 当前难度

未初始化时，主玩法页面和后端主流程接口会被拦截，避免绕过 onboarding 直接进入游戏流程。

### 2. 幸存者招募

- 支持单次招募幸存者
- 幸存者具有姓名、稀有度、职业和静态人格来源字段
- 已拥有的同名角色再次出现时，会转化为补偿材料
- 招募结果会生成入队记录，并写入招募日志
- 录屏路径下，重复补偿的呈现已做可读性强化

### 3. 角色展示与状态管理

当前角色系统分成两层：

- **静态层**：特质 / 作风 / 档案
- **动态层**：fatigue / health 派生出的当前状态标签

其中：

- `fatigue` 体现连续值勤的消耗
- `health` 体现高风险任务造成的伤势与压力
- 当前状态会以更容易被玩家理解的文案和标签形式展示

角色卡片当前重点展示：

- 角色姓名
- 稀有度
- 职业
- 当前状态
- 静态特质
- 作风短句
- 档案短句

### 4. 值勤派遣与结果反馈

当前值勤类型包括：

- 外出搜集
- 发电维护
- 库存炊事
- 夜间守卫
- 休整

值勤结算后，前端会把以下内容整合进同一个结果模块中展示：

- 叙事结果句
- 资源变化（如食物、电力、材料增减）
- 幸存者状态变化（疲劳、健康）

这样玩家执行一次操作后，可以立即看懂这次值勤“赚了什么、亏了什么、角色付出了什么”。

### 5. 应急补给协议

项目保留了一个条件触发的救急型商业化件：**战备应急补给协议**。

它不是常驻商城，而是在避难所进入压力状态后出现。当前触发逻辑会综合考虑资源压力与队伍状态压力。购买后会提供资源补给，并恢复部分幸存者状态。

当前保留了完整日志链路：

- `exposed`
- `closed`
- `purchased`

同时补了一层 suppress 节奏控制：

- 玩家点击暂缓后，会进入按行动次数计算的 suppress 窗口
- 普通压力不会在下一次小幅波动后立刻重新弹出
- 如果局面恶化到更严重状态，协议仍然可以越过 suppress 重新出现

### 6. 日志系统

当前日志主要分三类：

- 招募日志
- 值勤日志
- 补给日志

日志页已经做过收口，默认只展示每个分区的最新记录，更早内容可展开查看，以减少页面长度并提高演示时的可读性。

### 7. 本地调试与录屏支持

当前本地开发环境支持：

- `reset/new run`，便于重新进入 onboarding
- 应急补给协议测试状态重置
- 录屏模式开关（默认关闭，仅本地开发可见）
- 录屏模式开启后，可用于稳定复现演示路径

录屏模式用于本地演示，不属于正式默认玩法逻辑。

## 当前版本的演示路径

当前版本已经按录屏脚本做过一次收口，最适合展示的路径是：

```text
进入 -> onboarding -> 抽到 SSR
-> 抽到重复角色看到补偿

-> 疲劳与资源变化
-> offer 出现并购买
-> 日志里看到完整链路
```

这条路径的作用不是演一个功能清单，而是用一段连续操作把项目想表达的系统意图讲清楚：角色获取、资源循环、状态代价、压力触发、补给干预、日志闭环。

## 技术结构

- 前端：原生微信小程序
- 后端：Python 3.9、Flask、flask-cors
- 数据库：SQLite（Python 标准库 `sqlite3`）
- 本地开发：macOS、微信开发者工具、VS Code

技术路线保持轻量，重点服务于原型表达和作品集展示，不引入额外的重型前端框架或复杂工程化配置。

## 常用接口概览

- `GET /api/status`：检查后端是否运行
- `GET /api/init/status`：读取当前是否已完成游戏初始化
- `POST /api/init`：登记避难所代号、指挥官和难度，并写入开局资源
- `POST /api/dev/reset-init`：本地开发环境下重置当前局，重新进入 onboarding
- `GET /api/resources`：查看当前资源
- `POST /api/gacha`：执行一次幸存者招募
- `GET /api/survivors`：查看幸存者列表
- `POST /api/duty`：派遣幸存者执行值勤
- `POST /api/rest`：让幸存者休整
- `GET /api/gacha-logs`：查看招募日志
- `GET /api/duty-logs`：查看值勤日志
- `GET /api/emergency-offer/state`：查看补给协议当前状态
- `POST /api/emergency-offer/close`：暂缓处理补给协议
- `POST /api/emergency-offer/purchase`：购买补给协议
- `GET /api/emergency-offer/logs`：查看补给协议日志

说明：

- 未初始化时，除初始化相关接口外，主玩法接口会返回拦截结果
- 已完成初始化后，招募、值勤、日志和补给相关流程按正常逻辑运行
- 本地开发阶段支持 reset/new run 和录屏模式，便于重走 onboarding 与演示链路

## 本地运行方式

初始化或修复本地 SQLite 数据库：

```bash
python backend/init_db.py
```

启动 Flask 后端：

```bash
python backend/app.py
```

后端默认运行在：

```text
http://localhost:5001
```

然后在微信开发者工具中打开 `miniprogram/` 目录，并确认 `miniprogram/config.js` 中的接口地址指向本地 Flask 后端。

首次启动时，首页会先进入 onboarding。填写避难所代号、指挥官名字并选择难度后，才会正式进入首页、招募、值勤和日志流程。

## 后续规划（短期）

当前最明确的下一阶段目标，不是继续横向加功能，而是把项目推进成一个**7–10 天单局制、压力驱动的庇护所管理游戏 MVP**。

### 目标方向

把当前原型从：

- 可运行的系统循环

推进到：

- 有威胁弧线
- 有每日结算
- 有随机事件
- 有明确终局条件
- 有不可逆后果

### MVP 计划补的核心点

#### 1. 日推进 / 每日结算
引入“天数”和“每日固定行动次数”，让玩家不再处于无限循环里，而是在有限天数内做决策。每天结束时结算资源消耗、状态变化和外部威胁推进。

#### 2. 外部威胁弧线
加入一个明确的外部威胁倒计时，例如异变扩散、撤离窗口关闭、庇护所结构崩坏等。这样玩家的每一步操作都会带上“还能撑多久”的压力。

#### 3. 不可逆后果
当前状态管理已经有 `fatigue / health` 基础。后续会把健康归零后的代价从“数值变差”推进到更硬的局内后果，例如重伤离队、永久失去等，从而真正建立选择的重量。

#### 4. 随机事件层
在日结算或任务结算中加入小规模事件库，例如：
- 是否收留幸存者
- 是否冒险深入废墟
- 是否优先分配药物或食物

每个事件以二选一或多选一的方式给出后果，让系统从“结算型 demo”更像“真正的决策型游戏”。

#### 5. 胜利 / 失败条件
为单局引入明确终局，例如：
- 撑过第 7 天 / 第 10 天
- 修复通信塔
- 储备足够撤离物资
- 关键资源崩溃则失败

## 后续规划（中期）

如果单局制 MVP 跑通，后续最自然的方向不是推翻重做，而是在其上叠一层**局外成长**，把它逐步改造成长线游戏。

### 计划预留的长线方向

#### 1. player_meta / run_state 分层
后续会把“当前这一局”和“玩家长期档案”拆开：

- `run_state`：这一局的资源、天数、幸存者状态、当前事件
- `player_meta`：长期保留的解锁、图鉴、永久货币、成就

这样后面加长线内容时，不会把现有单局逻辑搅乱。

#### 2. 幸存者定义与实例分离
后续会将：

- `survivor_definitions`：静态角色定义
- `survivor_instances`：本局获得的角色实例

分开处理，为以后扩展这些内容做准备：

- 图鉴
- 永久解锁
- 多皮肤 / 多立绘
- 稀有度版本区分
- 局外收集系统

#### 3. 配置化事件
后续事件会尽量配置化，而不是继续把逻辑写死在流程里，便于持续补内容和做版本更新。

#### 4. 局外成长
中期最适合叠加的长线内容包括：

- 新职业解锁
- 新幸存者池
- 新事件分支
- 图鉴
- 永久设施等级
- 基地科技树
- 声望或阵营系统

## 角色画像与素材规划

当前项目后续会补一批统一风格的幸存者画像，用于：

- 招募结果页
- 首页幸存者卡
- 值勤页角色卡

### 画像方向

优先做 6–8 张半身像 / 胸像，覆盖核心职业与稀有度层级：

- SSR：2 张
- SR：2–3 张
- R / N：2–3 张

风格方向：

- 废土
- 低饱和
- 偏写实插画
- 功能性服装
- 背景简洁
- 适合裁切成卡牌头像

### 素材接入预留

后续接入时，考虑采用：

- `backend/static/portraits/`：存放图片资源
- `backend/data/survivor_portraits.json`：存放 `portrait_key -> portrait_url` 映射
- `survivor_definitions.json`：角色静态定义中预留 `portrait_key`

这样后续无论是继续做 MVP，还是往长线游戏扩展，都不会被素材接入方式卡住。
