# 数值阈值说明

本文档记录《废土避难所值勤管理器》当前 demo 版本的压力阈值，方便后续评审和调参。

当前项目是作品集原型，不是商业化完整数值版本。本文只记录现状和建议范围，不修改任何后端逻辑。

## 设计意图

- 资源压力不应该在普通开局状态立刻触发。标准难度初始资源是食物 80、电力 80、材料 55，均高于后端应急补给触发线。
- 队伍压力应该来自连续值勤后的疲劳和健康累积，而不是刚招募后立即触发。
- 应急补给应该像“救援支持”，只在资源、队伍状态或行动效率真的承压时出现，而不是固定商店弹窗。
- 关闭应急补给后应有短暂抑制，避免玩家每次回首页都被同一弹窗打断。

## 当前阈值总览

### 初始资源

定义位置：`backend/app.py` 的 `DIFFICULTY_STARTING_RESOURCES`。

| 难度 | 食物 | 电力 | 材料 | 招募券 |
| --- | ---: | ---: | ---: | ---: |
| 稳健 | 100 | 95 | 70 | 45 |
| 标准 | 80 | 80 | 55 | 30 |
| 极端 | 55 | 60 | 32 | 15 |

说明：后端资源压力触发线低于三档初始资源，因此正常开局不会立刻触发应急补给。

### 资源压力

定义位置：`backend/app.py` 的 `RESOURCE_LOW_THRESHOLDS` 和 `get_resource_pressure_reasons()`。

| 资源 | 当前触发条件 | 触发原因 |
| --- | ---: | --- |
| 食物 | `food <= 45` | `resource_food_low` |
| 电力 | `power <= 45` | `resource_power_low` |
| 材料 | `materials <= 15` | `resource_materials_low` |

严重压力定义位置：`backend/app.py` 的 `SEVERE_RESOURCE_THRESHOLDS` 和 `is_severe_emergency_pressure()`。

| 资源 | 当前严重压力条件 |
| --- | ---: |
| 食物 | `food <= 25` |
| 电力 | `power <= 25` |
| 材料 | `materials <= 6` |

严重压力会绕过关闭补给后的抑制，允许应急补给重新出现。

### 前端资源展示阈值

定义位置：`miniprogram/pages/home/home.js` 的 `RESOURCE_THRESHOLDS`。

| 资源 | 前端警戒 | 前端临界 |
| --- | ---: | ---: |
| 食物 | 75 | 60 |
| 电力 | 75 | 60 |
| 材料 | 35 | 25 |

注意：这些是首页视觉提示阈值，不是后端应急补给触发阈值。当前它们比后端触发线更早，适合 demo 中提前提示玩家关注资源变化。

### 疲劳与健康压力

定义位置：`backend/app.py` 的 `TEAM_*` 常量和 `get_team_state_pressure_reason()`。

| 类型 | 当前条件 | 说明 |
| --- | ---: | --- |
| 疲劳压力 | 至少 2 名 active 幸存者 `fatigue >= 80` | 返回 `survivor_fatigue_pressure` |
| 疲劳预警压力 | 至少 2 名 active 幸存者 `fatigue >= 75` | 作为较早的队伍压力判断 |
| 健康压力 | 至少 2 名 active 幸存者 `health <= 45` | 返回 `survivor_health_pressure` |
| 健康预警压力 | 至少 2 名 active 幸存者 `health <= 50` | 作为较早的队伍压力判断 |
| 最小队伍人数 | 2 | 由 `TEAM_PRESSURE_MIN_SURVIVORS` 控制 |

严重队伍压力定义位置：`backend/app.py` 的 `SEVERE_TEAM_HEALTH_THRESHOLD`、`SEVERE_TEAM_FATIGUE_THRESHOLD` 和 `is_severe_emergency_pressure()`。

| 类型 | 当前严重压力条件 |
| --- | ---: |
| 健康严重压力 | 任意非离队幸存者 `health <= 25` |
| 疲劳严重压力 | 至少 2 名非离队幸存者 `fatigue >= 92` |

状态标签定义位置：`backend/app.py` 的 `build_survivor_state_tags()`。

| 状态标签 | 当前条件 |
| --- | ---: |
| 重伤 | `health <= 30` |
| 状态不稳 | `health <= 60` |
| 疲惫 | `fatigue >= 80` |
| 状态良好 | 不满足以上条件 |

值勤状态惩罚定义位置：`backend/app.py` 的 `apply_survivor_state_penalty()`。

| 条件 | 当前效果 |
| --- | --- |
| `fatigue >= 80` | 正向资源收益乘以 0.75 |
| `health <= 30` | 正向资源收益乘以 0.75 |
| 两者同时满足 | 乘数叠加，约为 0.5625 |

### 值勤累积压力

定义位置：`backend/app.py`。

| 项目 | 当前值 | 定义名 |
| --- | ---: | --- |
| 每日行动次数 | 3 | `DEFAULT_ACTIONS_PER_DAY` |
| 搜集疲劳增加 | 18 | `FATIGUE_INCREASE_BY_DUTY["scavenge"]` |
| 发电疲劳增加 | 14 | `FATIGUE_INCREASE_BY_DUTY["generate_power"]` |
| 炊事疲劳增加 | 10 | `FATIGUE_INCREASE_BY_DUTY["cook"]` |
| 守卫疲劳增加 | 12 | `FATIGUE_INCREASE_BY_DUTY["guard"]` |
| 电力短缺全员疲劳惩罚 | 4 | `POWER_SHORTAGE_FATIGUE_PENALTY` |

高风险值勤定义位置：`backend/app.py` 的 `HIGH_RISK_DUTIES` 和 `resolve_high_risk_survivor_consequence()`。

| 条件 | 当前效果 |
| --- | --- |
| `health <= 10` 且执行搜集 / 守卫 | 禁止执行高风险值勤 |
| 搜集且随机命中 20% | 健康减少 5-12 |
| 守卫且随机命中 10% | 健康减少 3-8 |
| 执行高风险值勤前 `health <= 35` 或 `fatigue >= 85` | 重伤停工 2 天 |
| 执行高风险值勤前 `health <= 20` 且 `fatigue >= 85` | 离队 |

### 行动次数与低效触发

定义位置：`backend/app.py` 的 `get_action_pressure_reason()`。

| 项目 | 当前值 | 定义名 |
| --- | ---: | --- |
| 开始检查低效压力的非休整值勤次数 | 6 | `ACTION_PRESSURE_MIN_DUTY_COUNT` |
| 最近检查的值勤日志数量 | 4 | `RECENT_DUTY_LOG_LIMIT` |
| 低效日志数量阈值 | 3 | `LOW_EFFICIENCY_TRIGGER_COUNT` |
| 单次低效判断 | 食物变化 + 电力变化 + 材料变化 `<= 2` | `LOW_EFFICIENCY_TOTAL_CHANGE_THRESHOLD` |

解释：当非休整值勤总数至少达到 6 次后，系统检查最近 4 条非休整值勤日志；如果其中至少 3 条总资源变化不超过 2，就返回 `action_pressure`。

### 应急补给触发

定义位置：`backend/app.py` 的 `get_emergency_offer_trigger_reason()` 和 `get_emergency_offer_context()`。

触发来源包括：

- 资源压力：食物 / 电力 / 材料低于后端资源压力线。
- 队伍压力：至少 2 名幸存者疲劳或健康达到压力线。
- 行动压力：连续多次非休整值勤后，最近产出偏低。

触发优先级：

1. 如果资源、队伍、行动三类压力合计数量 `>= 2`，返回 `severe_multi_pressure`。
2. 否则优先返回第一个资源压力。
3. 如果没有资源压力，再返回队伍压力。
4. 如果没有资源和队伍压力，再返回行动压力。

补给真正显示还需要满足：

- 当前至少有 1 名未离队幸存者。
- 当前触发原因对应的补给没有购买过。
- 当前触发原因没有被关闭抑制。
- 触发原因不为空。

补给内容定义位置：`backend/app.py` 的 `EMERGENCY_OFFER`。

| 内容 | 当前值 |
| --- | ---: |
| 食物 | +25 |
| 电力 | +25 |
| 材料 | +15 |
| 招募券 | +2 |
| 全员疲劳恢复 | -10 |
| 全员健康恢复 | +3 |

### 关闭补给后的抑制

定义位置：`backend/app.py` 的 `EMERGENCY_OFFER_SUPPRESS_*`、`set_offer_suppression_window()` 和 `get_offer_suppression_status()`。

| 项目 | 当前值 | 说明 |
| --- | ---: | --- |
| 按行动数抑制 | 3 | 同一触发原因关闭后，接下来 3 次非休整值勤计数窗口内不再弹出 |
| 按天数抑制 | 1 | 关闭当天和下一天会被视为抑制窗口 |
| 严重压力覆盖 | 开启 | 严重资源或队伍压力会绕过抑制 |

关闭补给会写入 `offer_logs`，同时记录当时的 `action_count`。后续如果同一触发原因还在，但没有达到严重压力，会先看天数抑制和行动数抑制。

### 随机事件配置

定义位置：`backend/data/event_definitions.json`。

该文件记录随机事件选项的资源变化和幸存者状态变化，例如材料、电力、疲劳、健康、重伤停工或离队效果。它不定义应急补给触发阈值，但会间接改变资源和幸存者状态，从而影响是否触发压力。

## 推荐调参范围

以下只是保守建议范围，不代表当前代码需要修改。

| 项目 | 当前值 | 建议范围 | 调参方向 |
| --- | ---: | ---: | --- |
| 食物 / 电力压力线 | 45 | 35-55 | 越高越早触发补给 |
| 材料压力线 | 15 | 10-25 | 材料初始值较低，建议谨慎上调 |
| 食物 / 电力严重线 | 25 | 15-30 | 用于绕过关闭抑制 |
| 材料严重线 | 6 | 4-10 | 过高会让材料压力太频繁 |
| 疲劳压力线 | 80 | 75-85 | 越低越早触发队伍压力 |
| 疲劳预警线 | 75 | 70-80 | 建议低于正式压力线 5-10 点 |
| 健康压力线 | 45 | 40-55 | 越高越早触发医疗压力 |
| 健康预警线 | 50 | 45-60 | 可与前端“状态不稳”保持区分 |
| 严重健康线 | 25 | 20-30 | 用于救援兜底 |
| 严重疲劳线 | 92 | 88-95 | 防止极端疲劳被关闭抑制挡住 |
| 行动压力最小值勤数 | 6 | 5-8 | 太低会像固定弹窗 |
| 最近日志窗口 | 4 | 3-5 | 窗口越大越稳定但反馈更慢 |
| 低效日志数量 | 3 | 2-3 | 2 更敏感，3 更保守 |
| 单次低效阈值 | 2 | 0-3 | 越高越容易判定低效 |
| 关闭后行动抑制 | 3 | 2-4 | 太短会反复出现 |
| 关闭后天数抑制 | 1 | 1-2 | demo 建议保持短窗口 |
| 补给食物 / 电力 | 25 | 20-30 | 应能缓解但不应直接清空压力 |
| 补给材料 | 15 | 10-20 | 材料压力线低，补给不宜过大 |
| 补给疲劳恢复 | 10 | 8-15 | 应体现救援感 |
| 补给健康恢复 | 3 | 2-5 | 建议保持轻量恢复 |

## 看起来像 demo 占位值的部分

- `price_label` 当前是文本 `"¥6"`，没有真实支付接入，适合作品集演示但不是商业化定价。
- 前端首页资源展示阈值和后端补给触发阈值不一致：前端更早提示风险，后端更晚触发补给。当前适合 demo 引导，但后续需要统一设计口径。
- 行动压力用“最近日志总资源变化 <= 2”判断低效，规则直观但比较粗糙，后续可以按 duty 类型分别判断。
- 补给奖励固定为食物 +25 / 电力 +25 / 材料 +15，没有随难度、天数或压力来源动态变化。
- 关闭补给后的“天数抑制 + 行动数抑制”组合比较实用，但仍是 demo 级规则，需要结合实际测试频率再调。

## 手动测试清单

1. 启动后端并初始化数据库。

```bash
cd /Users/rayna/Documents/job/resume_game/wasteland-shelter-demo
source .venv/bin/activate
python backend/init_db.py
python backend/app.py
```

2. 在微信开发者工具中打开 `miniprogram/`，开始新一轮。
3. 选择“标准”难度，确认初始资源不会立刻触发应急补给。
4. 招募至少 1 名幸存者。
5. 连续执行多次值勤，观察资源变化、疲劳、健康和行动次数。
6. 如果出现随机事件，先完成事件选择，再继续值勤。
7. 观察首页资源提示：前端警戒可能比后端补给更早出现，这是当前设计。
8. 当食物或电力接近 45、材料接近 15 时，确认应急补给开始出现。
9. 当至少 2 名幸存者疲劳接近 75-80，或健康接近 45-50 时，确认队伍压力能触发补给。
10. 连续执行 6 次以上非休整值勤，观察最近 4 次产出偏低时是否出现行动压力补给。
11. 点击“暂缓处理”，确认补给不会立刻反复弹出。
12. 继续值勤 3 次左右，或推进到下一天后，再观察同一压力是否恢复显示。
13. 将资源或队伍状态压到严重压力附近，确认严重压力可以绕过关闭抑制。
