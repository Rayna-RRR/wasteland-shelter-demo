# 值勤页游戏感优化第 2 步

## 本次改动

本次只改小程序前端的值勤页，不改后端结算逻辑、不改数据库结构、不新增玩法系统。

主要变化：

- 将值勤页流程调整为“派遣名单 -> 派遣确认 -> 值勤结果”。
- 幸存者列表继续使用卡片选择，并强化选中态。
- 派遣按钮改成任务卡样式，保留原有值勤类型。
- 当前选择区域新增轻量风险提示。
- 值勤结果区域继续展示角色、任务、资源变化、疲劳/健康变化和结果文本。

## 选中状态如何工作

值勤页仍然使用原来的 `selectedSurvivorId` 保存当前选中的幸存者。

`applySurvivorSelection()` 会遍历幸存者列表：

- 当前选中角色追加 `survivor-card--selected` 样式。
- 当前选中角色增加 `selected: true`，用于显示“已选择”标记。
- 如果没有手动选择，会优先选择第一个可值勤成员。

这只影响前端展示，不改变幸存者数据结构和后端字段。

## 风险提示如何派生

风险提示来自前端已有字段：

- `fatigue`
- `health`
- `status`
- `unavailable`
- `statusNote`

判断规则：

- 不可值勤：显示不可派遣原因。
- 健康低于等于 30：提示继续值勤可能导致重伤停工。
- 健康低于等于 60：提示建议避免高风险任务。
- 疲劳高于等于 80：提示继续值勤可能加重状态。
- 疲劳高于等于 60：提示建议低风险任务或休整。
- 其他情况：提示当前状态可继续执行普通值勤。

这些判断只用于界面提示，不参与后端结算。

## 派遣 API 兼容性

值勤派遣仍然调用：

```js
api.assignDuty({
  survivor_id: survivorId,
  duty_type: dutyType
})
```

休整仍然调用：

```js
api.restSurvivor({
  survivor_id: survivorId
})
```

因此后端无需改动。
