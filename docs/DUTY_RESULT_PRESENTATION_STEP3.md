# Duty Result Presentation Step 3

## 目标

本步骤把 Duty 页的值勤反馈从单句提示升级为紧凑的避难所行动结算报告。改动只发生在前端展示层，不改变后端值勤结算、幸存者状态计算、资源计算或数据库结构。

## 结果卡现在显示什么

- 结算标题：显示“值勤完成 · 任务类型”。
- 幸存者摘要：显示头像、姓名、稀有度、职能和当前状态标签。
- 任务摘要：值勤任务会在标题区显示；如果任务已有图标资产，则同步显示任务图标。
- 行动描述：继续使用后端返回的 `result_text`，保留原有结算文案。
- 资源变化：以紧凑 chip 展示食物、电力、材料、招募券等实际变化项。
- 幸存者状态变化：以紧凑 chip 展示疲劳和健康变化。
- 下一步建议：根据本次结算结果给出短句提示，帮助玩家判断下一次行动。

## 资源变化展示

前端读取后端已有字段：

- `food_change`
- `power_change`
- `materials_change`
- `premium_currency_change`（如果存在）

值为 0 的资源不展示，避免卡片变成调试面板。正向变化使用收益色，负向变化使用风险色。

## 健康 / 疲劳变化展示

前端优先读取 `result.survivor_state` 或 `result.survivor_state_change` 中的：

- `fatigue_change`
- `health_change`

如果字段缺失，则用值勤前后幸存者状态差值兜底。疲劳上升视为状态成本，健康下降视为风险。

## 严重度标签规则

严重度只用于展示，不影响后端结算。

- `高风险结果`：出现重伤 / 离队后果，或健康大幅下降，或结算后健康过低。
- `健康风险`：健康下降明显，或结算后健康偏低。
- `疲劳上升`：疲劳增加明显，或结算后疲劳很高。
- `资源承压`：资源消耗较高，或收益很低且存在消耗。
- `收益良好`：资源收益明显为正。
- `状态可控`：未命中以上风险或高收益条件时的安全兜底。

## 下一步建议规则

下一步建议基于展示层已有数据生成：

- 健康风险优先提示避免继续高风险行动。
- 疲劳较高时提示安排休整。
- 食物、电力、材料有收益时提示对应压力缓解。
- 收益有限或资源承压时提示调整派遣对象或任务类型。
- 休整产生恢复时提示重新评估低风险任务。

## 队伍状态反馈

Duty 页会基于当前幸存者列表生成一个紧凑的队伍状态提示，显示在派遣确认区域。

使用字段：

- `status`
- `assignable` / `unavailable`
- `fatigue`
- `health`

展示内容：

- 可派遣人数 / 当前队伍人数
- 高疲劳成员数量
- 健康风险成员数量
- 一句短提示

判断规则：

- 有成员健康很低，或低健康成员达到多个：显示 `健康风险`，提示避免继续高风险值勤。
- 多名成员疲劳很高：显示 `疲劳累积`，提示安排休整。
- 没有可派遣成员：显示 `暂不可派遣`。
- 其他情况：显示 `状态可控`。

该提示只用于前端可读性，不改变派遣、休整、资源或事件结算。

## 手动测试路径

1. 打开 Mini Program 的 Duty 页。
2. 从分组派遣名单中选择一名幸存者。
3. 选择不同任务类型并执行值勤。
4. 确认派遣确认区域显示队伍状态提示。
5. 确认任务选择卡和结果卡中的任务图标正常显示。
6. 确认结果卡显示幸存者头像、姓名、稀有度、职能和状态。
7. 确认资源变化以紧凑 chip 展示。
8. 确认疲劳和健康变化以紧凑 chip 展示。
9. 确认严重度标签与本次结果大致匹配。
10. 确认下一步建议出现且文案简短。
11. 确认资源刷新、幸存者刷新、应急补给触发和日志入口仍可正常使用。

## 作品集价值

这一步让值勤反馈更像游戏内结算报告，而不是原始调试信息。对 R&D-support / 游戏运营场景来说，玩家可以快速理解“为什么派这名幸存者、本次行动发生了什么、代价是什么、队伍是否正在变危险、下一步应该怎么做”；同时结算字段仍来自稳定后端数据，便于后续用日志复盘数值压力和玩家行为。
