# 应急补给展示优化第 2 步

## 当前补给卡片展示内容

首页的应急补给卡片现在展示：

- 补给协议名称。
- 协议状态。
- 应急补给图片。
- 系统压力说明。
- 压力来源。
- 协议消耗。
- 补给内容。
- 协议作用。
- 暂缓处理按钮。
- 启用协议按钮。

展示目标是让应急补给呈现为避难所压力下开放的救援协议，减少普通商业礼包感。

## 素材位置

应急补给图片存放在：

```text
miniprogram/assets/images/items/emergency_supply.png
```

页面中使用的小程序路径为：

```text
/assets/images/items/emergency_supply.png
```

该素材只接入首页应急补给卡片，没有复用到其他页面。

## 图片兜底

首页 `image` 组件绑定了加载失败处理。

如果图片缺失或加载失败：

- 前端会隐藏图片区域。
- 补给卡片文字、奖励内容和按钮继续显示。
- 不影响关闭或启用补给协议。

## 触发原因中文转换

前端通过 `miniprogram/utils/format.js` 把后端 `trigger_reason` 转成中文。

当前主要映射：

- `resource_food_low` -> `食物储备告急`
- `resource_power_low` -> `电力供应不稳`
- `resource_materials_low` -> `维修材料不足`
- `survivor_fatigue_pressure` -> `队伍疲劳累积`
- `survivor_health_pressure` -> `幸存者健康风险`
- `severe_multi_pressure` -> `多重压力叠加`
- `action_pressure` -> `连续行动后补给压力上升`

同时保留旧触发原因名称的显示兼容，避免旧日志显示成空白。

## 补给日志动作

日志页展示应急补给的三类记录：

- `exposed`：补给开放，表示系统检测到压力并开放补给协议。
- `closed`：玩家关闭，表示玩家选择暂缓处理，后端记录关闭并进入抑制窗口。
- `purchased`：补给购买，表示玩家启用付费补给协议，资源和队伍状态获得恢复。

每条日志会尽量展示：

- 操作类型。
- 压力来源。
- 触发节点。
- 触发时资源快照。
- 幸存者数量。
- 记录时间或记录编号。
- 处理结果。

如果某些字段缺失，前端显示 `--` 或 `未记录`，页面不会崩溃。

## 手动测试路径

1. 打开微信开发者工具。
2. 进入首页。
3. 通过资源压力或值勤行动触发应急补给。
4. 确认应急补给图片出现在补给卡片内。
5. 确认压力来源显示为中文，例如 `食物储备告急` 或 `多重压力叠加`。
6. 确认卡片包含补给名称、压力说明、补给内容、协议作用、暂缓处理按钮和启用协议按钮。
7. 点击暂缓处理，确认补给不会立刻反复刷屏。
8. 再次触发或启用补给协议，确认流程正常。
9. 打开日志页。
10. 确认 `补给开放`、`玩家关闭`、`补给购买` 记录可见。
11. 确认日志中的压力来源和资源快照可读。
12. 在没有补给日志时，确认页面显示轻量空状态。

## 产品与作品集价值

这一步把应急补给从“弹出一个补给卡”补齐为可解释、可追踪的完整功能：

- 触发原因能被玩家看懂。
- 玩家响应能通过日志复盘。
- 资源快照让运营和研发支持视角更容易判断触发是否合理。
- 图片和文案强化了避难所救援协议的语境，但没有新增商店页或复杂商业系统。
