# 运营与产品复盘说明

本文档面向 HR、游戏运营面试官、业务侧评审和研发支持运营岗位评审，用于说明《废土避难所值勤管理器》这个作品集项目的产品目标、系统循环和运营复盘价值。

它不是一款完整商业化游戏，而是一个轻量原型，用较小实现成本展示系统设计、压力设计、补给触发、日志记录和可复盘意识。

## 1. Project Goal

《废土避难所值勤管理器》是一个 WeChat Mini Program + Flask + SQLite 的废土避难所管理原型。

项目目标不是证明“已经完成商业化上线”，而是证明候选人能把一个小型游戏系统做成可运行、可解释、可复盘的闭环：

- 玩家能招募幸存者。
- 玩家能把幸存者分配到不同值勤岗位。
- 值勤会改变食物、电力、材料等资源。
- 幸存者会积累疲劳，健康也可能下降。
- 随机事件会打断流程，要求玩家做选择。
- 应急补给会根据压力触发，而不是固定商城入口。
- 招募、值勤和补给行为会写入 SQLite 日志，便于复盘。

从作品集角度，它展示的是：

- 规则设计能力。
- 系统循环搭建能力。
- 运营视角下的压力触发与行为记录意识。
- 轻量实现和交付能力。

## 2. Core Loop

当前核心循环可以概括为：

```text
Recruit survivor
-> Assign duty
-> Resources change
-> Fatigue / health changes
-> Random event may block actions until resolved
-> Emergency offer may be triggered
-> Logs are recorded for review
```

中文流程：

```text
招募幸存者
-> 分配值勤
-> 资源变化
-> 疲劳 / 健康变化
-> 随机事件可能阻断行动，直到玩家处理
-> 压力满足条件时可能触发应急补给
-> 日志记录行为和结果，便于复盘
```

这个循环的重点不是内容量，而是让每一步都能影响后续状态。招募带来队伍成员，值勤带来资源和状态变化，随机事件制造选择压力，应急补给回应真实压力，日志记录玩家和系统发生了什么。

## 3. Why fatigue and health matter

疲劳和健康让幸存者管理更像游戏系统，而不是简单的“点按钮获得资源”。

疲劳代表重复行动成本：

- 值勤会增加疲劳。
- 高疲劳会降低后续行动收益。
- 多名幸存者疲劳过高时，会形成队伍压力。
- 玩家需要考虑是否继续压榨某个角色，还是换人或休整。

健康代表风险和恢复压力：

- 高风险值勤可能造成健康下降。
- 低健康会提升受伤、停工或离队风险。
- 事件也可能影响健康或造成重伤停工。
- 玩家需要在资源收益和人员安全之间做取舍。

这两个状态让幸存者从静态卡牌变成需要管理的队伍资源。对运营和产品评审来说，它说明项目不仅有“资源数值”，也有“角色状态压力”和“风险反馈”。

更多阈值说明见 [BALANCE_THRESHOLDS.md](BALANCE_THRESHOLDS.md)。

## 4. Random Event Design

随机事件的作用是打断机械循环，让玩家在继续行动前处理当前避难所压力。

当前逻辑中，如果存在未处理事件：

- 招募和值勤行动会被阻止。
- 玩家必须先回到首页处理事件选择。
- 事件处理后，行动流程才恢复。

这样做的设计理由是：

- 事件不只是装饰文案，而是行动节奏的一部分。
- 玩家不能无视避难所正在发生的问题。
- 事件选择会改变资源、幸存者状态或长期结果。

当前事件池是 demo 级别，数量较少，主要用于证明机制：

- 资源交换：用一种资源换另一种资源。
- 队伍风险：让目标幸存者受伤、停工或离队。
- 避难所压力：表现滤水、电力、通道等设施问题。
- 支持选择：消耗资源避免更差后果。
- 不可逆后果：个别选择会导致幸存者离队。

更多事件配置说明见 [RANDOM_EVENTS.md](RANDOM_EVENTS.md)。

## 5. Emergency Offer Design

应急补给不是一个静态商城入口，而是压力驱动的支持机制。

当前补给日志记录三类行为：

- `exposed`：补给被展示给玩家。
- `closed`：玩家暂缓处理。
- `purchased`：玩家启用补给协议。

设计重点：

- 补给应该在资源、队伍状态或行动效率出现压力时出现。
- 它应该像“救援支持”，而不是一进页面就出现的固定售卖弹窗。
- 关闭补给后需要短暂抑制，避免同一压力反复弹出打断体验。
- 严重压力可以绕过关闭抑制，保证兜底救援感。

项目没有真实支付系统。当前 `¥6` 只是演示用价格文案，用于表达“商业化触发点如何和压力设计连接”，不代表已经接入支付。

## 6. Logging and Review Value

日志是这个项目面向运营 / 研发支持运营岗位的重要部分。

当前日志包括：

- Gacha logs：记录招募结果，包括幸存者、稀有度、职业和重复角色补偿。
- Duty logs：记录值勤角色、值勤类型、结果文本和资源变化。
- Offer logs：记录补给曝光、关闭、购买，以及触发原因和当时资源状态。

这些日志的复盘价值：

- 观察玩家是否频繁招募，还是更偏向派遣值勤。
- 观察哪些 duty 类型导致资源压力或队伍状态压力。
- 观察补给是被展示、关闭还是购买。
- 观察应急补给是否过早、过频繁或太晚出现。
- 为后续调参提供依据，而不是只凭主观感受判断。

当前日志不是完整数据分析后台，但已经体现了“行为记录 -> 状态结果 -> 运营复盘”的基本意识。

## 7. What this prototype proves

这个原型能证明的能力包括：

### Rule design

能把招募、值勤、资源、疲劳、健康、事件、补给和日志连接成一条可运行规则链。

### System thinking

不是孤立做页面，而是让每个模块影响其他模块：值勤影响资源和状态，状态影响事件和补给，补给又影响资源和恢复。

### R&D-support operations awareness

能从业务和研发之间的视角解释系统：

- 哪些规则可调。
- 哪些日志能帮助复盘。
- 哪些行为可能影响玩家体验。
- 哪些功能是 demo，哪些可以后续扩展。

### Data/logging awareness

核心行为不是只显示在界面上，也会被记录到 SQLite，便于检查和复盘。

### Lightweight implementation ability

项目使用原生小程序、Flask 和 SQLite，没有引入重框架。它适合作为短周期作品集，重点展示闭环完成度和可解释性。

## 8. Current limitations

当前限制需要诚实说明：

- 数值平衡是 demo 级别，还需要更多测试样本。
- 静态 Web Demo 只是浏览器预览，不连接 Flask / SQLite。
- 没有真实支付系统。
- 没有生产级 LLM agent，也没有运行时依赖 LLM。
- LLM 只适合作为未来可选叙事增强，不应进入核心结算。
- 没有多用户后端、账号系统或线上服务。
- 没有完整运营后台或自动化数据报表。
- 事件池和内容量较小，主要用于展示机制。

这些限制不影响作品集目标，因为项目重点是证明“轻量系统闭环”和“运营复盘意识”，不是宣称已经商业化完成。

## 9. Next possible improvements

后续可以优先做的小步改进：

- 调整资源、疲劳、健康和应急补给阈值，见 [BALANCE_THRESHOLDS.md](BALANCE_THRESHOLDS.md)。
- 继续维护轻量后端 smoke test：`scripts/smoke_test_backend.py`。
- 增加更直观的 replay / analytics summary，例如“一局中触发了几次补给、多少次值勤、资源最低点是多少”。
- 增加更多事件类别，例如天气、设备老化、疾病、内部争执和外部交易。
- 增加事件日志，便于评审时展示玩家选择和系统后果。
- 增加事件插图配置，把图文内容从前端映射逐步整理到配置层。
- 可选加入 LLM narrative helper，只用于生成特殊事件文案，不影响核心数值结算。

建议优先顺序：

1. 保持核心循环稳定。
2. 扩展复盘展示能力。
3. 再增加更多事件和内容。
4. 最后再考虑可选 LLM 文案增强。
