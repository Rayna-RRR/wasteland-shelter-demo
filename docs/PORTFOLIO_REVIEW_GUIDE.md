# Portfolio Review Guide

## 先打开什么

如果只想快速了解项目效果，先打开静态 Web Demo：

```bash
open web-demo/index.html
```

如果想查看完整实现，再运行主项目：

1. 启动 Flask 后端。
2. 用微信开发者工具打开 `miniprogram/`。
3. 按 onboarding -> 招募 -> 值勤 -> 日志路径体验。

Web Demo 是浏览器预览；微信小程序 + Flask + SQLite 才是主实现。

## 建议体验的核心循环

建议按这条路径查看：

```text
招募幸存者 -> 选择幸存者 -> 派遣值勤 -> 查看资源变化和疲劳 / 健康变化 -> 触发随机事件或应急补给 -> 查看日志
```

重点观察：

- 幸存者是否有稀有度、职业和状态信息。
- 值勤结果是否清楚展示资源和状态变化。
- 队伍疲劳 / 健康压力是否能被玩家理解。
- 应急补给是否由压力触发。
- 日志是否能回看 exposed / closed / purchased 等关键行为。

## 体现 game operations / R&D-support 思考的部分

- 把玩家动作、资源压力、队伍状态和结果反馈串成闭环。
- 用日志记录招募、值勤和补给行为，方便复盘系统表现。
- 让应急补给基于压力出现，而不是作为随机商城弹窗。
- 用静态 Web Demo 降低作品集评审门槛，同时保留小程序主实现的完整逻辑。
- 用视觉素材提升可读性，但不把素材当作系统设计的替代品。

## 有意简化的部分

- 数值平衡是 demo 级别，主要用于表达压力循环。
- Web Demo 使用 mock state，不连接 Flask 或 SQLite。
- 没有账号、支付、线上服务或完整运营后台。
- LLM 不在核心运行时逻辑中。
- 随机事件和应急补给仍是轻量演示规模。

## 评审建议

如果时间只有 1 分钟：

1. 先看 README 的 Quick Preview 和 Core Loop。
2. 打开 `web-demo/index.html` 快速操作一次招募和值勤。
3. 再看小程序页面结构和后端接口，确认主实现不是纯静态页面。

如果时间有 5–10 分钟：

1. 跑主小程序原型。
2. 完成 onboarding。
3. 招募幸存者。
4. 派遣高风险值勤。
5. 查看应急补给和日志链路。
